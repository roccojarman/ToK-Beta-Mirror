# Mirror Protocol

Version: 0.1.0
Status: provisional operating protocol

## Purpose

The Mirror is a governed working reflection in which received material can be preserved, digested, extrapolated, related, classified, and prepared for explicit promotion without confusing provisional understanding with canonical content.

This repository-native protocol is the governing specification. A later MCP server or application may implement it but must not silently replace it.

## States

`received -> digested -> extrapolated -> related -> classified -> positioned -> assessed -> ratified -> promoted -> impressed`

A candidate may move backwards when revised. Ratification and promotion are explicit events, not conclusions inferred from earlier states.

## Operations

| Operation | Effect |
|---|---|
| receive | Preserve supplied material and provenance |
| extrapolate | Produce separately attributed candidate principles, terms, relations, or questions |
| relate | Propose typed correspondences without merging identities |
| classify | Propose an entry kind from the maintained Taxonomy |
| query | Retrieve material under explicit jurisdiction and metadata filters |
| assess-readiness | Report evidence, omissions, and uncertainty |
| propose-graft | Propose a new placement while preserving identity |
| propose-prune | Propose removal of a placement or relation |
| propose-promotion | Prepare a destination-specific reviewed change |
| ratify | Record authorised human approval |
| re-impress | Rebuild WholeTree from canonical sources through its workflow |

## Invariants

- Source and extrapolation remain distinguishable.
- Content, General Logia, and Librarian Logia remain independently filterable.
- Stable identity survives structural edits.
- Agents may propose and assess. They do not self-ratify.
- Every canonical change has provenance, authority, and repository history.
- Unknowns remain explicit.
